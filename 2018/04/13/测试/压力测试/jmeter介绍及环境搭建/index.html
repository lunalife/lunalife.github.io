Jmeter介绍及环境搭建

简介

Apache JMeter是Apache组织开发的基于Java的压力测试工具。用于对软件做压力测试，它最初被设计用于Web应用测试，但后来扩展到其他测试领域。
–百度百科

Jmeter使用了不同技术和协议，是一款可以进行配置和执行负载测试、性能测试和压力测试的工具。

它能够模拟不同类型的请求，例如各种类型的数据库，FTP, HTTP, HTTPS 或者其他的服务端应用。

  1. 负载测试: 这类测试使系统或者应用程序在预先设计好的极端场景下测试运行。这类测试用来评估系统或者程序在极端条件下的行为。

  2. 性能测试: 这种测试被用来检测系统的性能表现，包括特定情况下，系统的响应能力和稳定性。

  3. 压力测试: 这类测试通过载入更多的外部资源，并使系统组件超越其所设定的能力范围，试图使系统挂掉。

工作原理

JMeter作为服务器与浏览器之间的代理网关，捕获中间的交互信息。通过线程组模拟真实用户对web服务器的访问压力，与LR的工作原理基本一致。

线程组/用户 Threads/Users

线程组是所有测试计划的根节点，所有的controller和sample都要在Thread Group下，其它的elements,如listener必须直接放在test plan下面，保证能作用于所有的thread group。

一个性能测试请求负载是基于一个线程组完成的。一个测试计划必须有一个线程组。

如上界面，有三个添加线程组的选项，名字不一样， 创建之后，其界面是完全一样的。

  1. thread group(线程组)：这个就是我们通常添加运行的线程。通俗的讲一个线程组,可以看做一个虚拟用户组，线程组中的每个线程都可以理解为一个虚拟用户。线程组中包含的线程数量在测试执行过程中是不会发生改变的。
  2. setup thread group:用于执行预测试操作。这些类型的线程执行测试前进行定期线程组的执行。相当于setup。
  3. teardown thread group: 用于执行测试后动作。这些类型的线程执行测试结束后执行定期的线程组。相当于teardown。

每个线程组将执行完整的测试计划,完全独立于其他测试线程。多线程用于模拟并发连接到您的服务器应用程序。

过渡时期ramp-up:所有线程完成所需要的时间，如果一共有10个线程，ramp-up的时间设置为100秒，每个线程10秒的运行时间。过渡需要足够长的时间,以避免太大工作量的一个测试。

Test Fragment 测试片段


测试片段的元素是一种特殊类型的控制器存在于测试计划树中和线程组元素在同一个层级。它和线程组不同的是它不能被执行,除非它是通过Module Controller或者Include_Controller引用的。

Config Element配置元件


配置元件（config element）用于提供对静态数据配置的支持。
CSV Data Set config 可以将本地数据文件形成数据池（Data Pool），而对应于HTTP Request Sampler和 TCP Request Sampler等类型的配制无件则可以修改Sampler的默认数据。（例如，HTTP Cookie Manager 可以用于对 HTTP Request Sampler 的cookie 进行管理）

Controller 控制器

包括取样器(sample)和逻辑控制器(Logic Controller)，用这些元件来处理测试。

3.1 取样器


取样器（Sample）是性能测试中向服务器发送请求，记录响应信息，记录响应时间的最小单元，他们按照在tree中出现的顺序执行，JMeter 原生支持多种不同的sampler,如下：

  1. FTP Request
  2. HTTP Request
  3. JDBC Request
  4. Java object request
  5. LDAP Request
  6. SOAP/XML-RPC Request
  7. WebService (SOAP) Request

每个sampler都有几个属性需要设置，您可以进一步定制一个取样器通过添加一个或多个配置元素到测试计划。每种不同类型的 sampler 可以根据设置的参数向服务器发出不同类型的请求。

例如：
添加一个HTTP Request Sampler发送一个HTTP请求. 您还可以定制一个请求通过添加一个或多个取样器的配置元素。
如果你要发送多个相同类型的请求(例如,HTTP请求)到同一台服务器上,考虑使用一个默认的配置Element。每个控制器都有一个或多个默认Element。
记得添加一个侦听器到您的测试计划来查看你的请求和/或将结果存储到磁盘上。

如果你有兴趣让JMeter在响应你的请求时执行基本的验证,添加一个assertion到sampler。

例如，
压力测试一个web app，服务器返回”HTTP Response” code，页面可能有错误或缺失的部分。
可以添加assert检查特定的HTML标记,常见的字符串错误，等等。
JMeter允许使用正则表达式创建assert。

3.2 逻辑控制器


逻辑控制器可以改变来自他们的子元素的请求顺序。他们可以修改请求,使JMeter重复请求,等等。
包括两类元件：
一类是用于控制测试计划中 sampler 节点发送请求的逻辑顺序的控制器，常用的有

  • 如果（If）控制器
  • switch Controller
  • Runtime Controller
  • 循环控制器等

另一类是用来组织可控制 sampler 来节点的，如

  • 事务控制器
  • 吞吐量控制器。

要了解逻辑控制器在测试计划的影响,思考以下的测试树(步骤):

  1. Test Plan
  2. Thread Group
  3. Once Only Controller
  4. 登录请求 (an HTTP Request)
  5. Load Search Page (HTTP Sampler)
  6. Interleave Controller
  7. Search “A” (HTTP Sampler)
  8. Search “B” (HTTP Sampler)
  9. HTTP default request (Configuration Element)
  10. HTTP default request (Configuration Element)
  11. Cookie Manager (Configuration Element)

解释：
1-2 创建测试计划和线程组
3-4 第一件事是执行登录请求。后续的迭代将跳过该请求，Once Only Controller只执行一次。
5 在登录之后,下一个sampler加载搜索页面，这里无任何逻辑控制器过滤逻辑。
6-8 搜索动作。我们想重新加载页面来搜索这两个页面。使用interleave Controller一次传递一个child request。
9-10 tree中的下一个元素是另一个HTTP defalt request
11 最后一个element是HTTP Cookie Manager，它应该被添加到所有的web test中，否则JMeter会忽略cookie。

定时器Timers

定时器（Timer）用于操作之间设置等待时间，默认情况下,一个JMeter线程发送每个请求之间没有停顿。JMeter 定义了Bean Shell Timer、Constant Throughput Timer、固定定时器等不同类型的Timer。

建议您通过添加一个timers到线程组来指定一个延迟计时器。如果不添加延时器，JMeter可能压垮你的服务器因为会在很短的时间发送太多的请求。

在每个取样器之前，计时器会使JMeter延迟一定时间。如果添加多个定时器到线程组中,JMeter会计算计时器的总和,执行应用了timer的sampler之前会暂停相应的时间。Timers可以作为sampler或者controller的children添加，来限制应用到哪一个sampler上面。在测试计划中，提供一个暂停可以使用 Test Action Sampler.

断言Assertions

assertions允许你插入期望数据关于从服务器接收到响应的测试的返回值。测试你的应用程序返回值和你期望的结果。

例如,您可以assert一个查询的响应必须包含一些特定的文本。你指定的文本可以是Perl风格的正则表达式，可以指定响应包含的text或者匹配整个响应。

你可以添加assertion到任何的sampler，例如，添加assertion到HTTP Request来检查text，”“，JMeter将检查文本出现在HTTP响应。如果JMeter无法找到文本,那么它将标志着这是一个失败的请求。

注意,断言适用于所有sampler的范围。添加assert为某个sampler的chirld来应用于这个sampler。

查看assertion的结果,添加一个Assertion Listener到Thread Group.失败的assertion也会出现在树视图和表监听中，会总结到error统计百分比页面，如总结报告。

11 Properties and Variables

JMeter的properties定义在jmeter.properties文件中的。jmeter属性是全局的,主要是用于定义一些jmeter使用默认值。例如，remote_hosts定义将尝试远程运行了的服务器。properties可以在测试计划中引用，但是不能用于线程专用的值。每个线程的值可能是相同的，也可能不同。如果一个变量被一个线程更新,这里只改变线程变量的副本

6 关于登录

有的网站需要登录后才能进行后续的操作，在web浏览器中，将显示为登录用户名和密码和一个按钮的表单。提交按钮会发送一个post请求，将表单的值作为参数发送。
JMeter中操作需要先添加一个HTTP Request，发送一个POST请求，你需要知道表单中字段的名称，然后添加目标页面。可以查看页面的源代码找到字段名称。(如果这是很难做到,您可以使用JMeter代理记录器来记录登录序列),设置提交按钮的目标path。单击Add按钮两次,输入用户名和密码信息。有时登录表单中会包含隐藏字段这些字段也需要添加。

url重写处理用户session

如果您的web应用程序使用URL重写而不是Cookie保存会话信息的话,那么你就需要做一些额外的工作来测试你的网站。为了正确应对URL重写,JMeter需要解析从服务器接收到的HTML并检索唯一的session ID。使用合适的HTTP URL Re-writing Modifier来完成这个。只需输入您的会话ID参数的名称到modifier中,会找到它并将它添加到每个请求中。如果请求已经有一个值,它将被取代。如果Cache Session Id已经被检查，前面的HTTP sampler没有包含session id,则最后一个会话id将被保存并且使用。
